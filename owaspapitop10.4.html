About the OWASP Top 10

The OWASP Top 10 is the reference standard for the most critical web application security risks. It represents the consensus of security experts worldwide and is updated periodically to reflect the evolving threat landscape.

The 2021 edition is built from two primary data sources: contributions from over 40 data providers analyzing vulnerabilities across more than 512,000 applications, and a comprehensive industry survey of security experts. This data-driven approach ensures the Top 10 reflects real-world risks.

Organizations that incorporate the OWASP Top 10 into their security programs can reduce their vulnerability risk by up to 60% when implemented as part of a comprehensive security strategy.

Understanding Web Application Security

What is Web Application Security?

Web application security focuses on protecting websites and online services against various security threats that exploit vulnerabilities in an application's code or underlying systems.

  • Protects sensitive data from unauthorized access
  • Ensures application functionality works as intended
  • Maintains availability of services to legitimate users

Why is it important?

With the growing complexity of web applications and increasing value of data they process, understanding and mitigating security risks is essential to protect users, data, and maintain trust.

New Categories

  • A4

    Insecure Design

    A new category focusing on flaws in the design phase, rather than implementation bugs. This represents a shift toward "security by design" principles.

  • A8

    Software and Data Integrity Failures

    Focuses on software updates, critical data, and CI/CD pipelines without integrity verification. Includes and expands on insecure deserialization.

  • A10

    Server-Side Request Forgery

    A critical vulnerability where attackers can make the server fetch resources, potentially accessing internal systems or sensitive cloud metadata.

Key Insight: These new categories reflect the changing landscape of web application architecture, with more complex supply chains, cloud infrastructure, and API ecosystems.

Renamed & Refocused

  • A2

    Cryptographic Failures

    Renamed from "Sensitive Data Exposure" to focus on the root cause (cryptographic issues) rather than the symptom (data exposure).

  • A3

    Injection

    Now includes all injection vulnerabilities including SQL, NoSQL, OS command, ORM, LDAP, and notably Cross-Site Scripting (XSS).

  • A7

    Identification and Authentication Failures

    Expanded from "Broken Authentication" to include broader identification issues, session management, and other authentication-related flaws.

Key Insight: These changes reflect a more mature understanding of the underlying causes of vulnerabilities, moving away from symptoms toward root causes.

Major Position Changes

  • A1

    Broken Access Control 4

    Moved from #5 to #1, reflecting the increasing prevalence of access control failures in complex permission models and API-driven applications.

  • A3

    Injection 2

    Dropped from #1 to #3, likely due to widespread adoption of frameworks with built-in protections against common injection flaws.

  • A6

    Vulnerable Components 3

    Moved up from #9 to #6, highlighting the increasing risk of dependency-based vulnerabilities in modern web applications.

Key Insight: The rise of Broken Access Control to #1 signals the challenges of implementing proper authorization in complex, API-driven applications with many different user roles and permissions.

Essential Security Best Practices

Authentication & Access Control

Implementing robust authentication and access control is critical for protecting sensitive resources and operations.

Multi-Factor Authentication

Implement MFA for all privileged users and sensitive operations to add an additional layer of security beyond passwords.

Implementation Options:
  • Time-based One-Time Passwords (TOTP)
  • Push notifications to mobile devices
  • Biometric authentication (when available)
  • Hardware security keys (U2F/FIDO2)

Secure Session Management

Properly implemented session management prevents session hijacking and fixation attacks.

Requirements:
  • Regenerate session IDs after authentication
  • Use secure, HttpOnly, SameSite cookies
  • Implement reasonable timeout periods
  • Provide secure logout functionality

JWT Security Best Practices

Critical Implementation Requirements:
  • Use strong signing algorithms (RS256, ES256)
  • Include expiration (exp), issued-at (iat) claims
  • Add audience (aud) and issuer (iss) claims
  • Keep tokens short-lived (15-60 minutes)
  • Implement token revocation mechanism
Security Vulnerabilities to Avoid:
  • Using "none" algorithm or weak HMAC keys
  • Accepting unsigned/unverified tokens
  • Storing sensitive data in the payload
  • Missing proper validation of all claims
  • Using excessively long-lived tokens

Node.js Authentication Implementation (Express.js)

const jwt = require('jsonwebtoken');
const bcrypt = require('bcrypt');
const { body, validationResult } = require('express-validator');

// Password validation rules
const passwordRules = [
  body('password')
    .isLength({ min: 12 }).withMessage('Password must be at least 12 characters')
    .matches(/[a-z]/).withMessage('Password must contain lowercase letter')
    .matches(/[A-Z]/).withMessage('Password must contain uppercase letter')
    .matches(/[0-9]/).withMessage('Password must contain a number')
    .matches(/[^a-zA-Z0-9]/).withMessage('Password must contain a special character')
];

// User authentication
async function authenticateUser(req, res) {
  try {
    // Input validation
    const errors = validationResult(req);
    if (!errors.isEmpty()) {
      return res.status(400).json({ errors: errors.array() });
    }
    
    const { username, password } = req.body;
    
    // Find user - in a real app, use a database query
    const user = await User.findOne({ username });
    
    if (!user) {
      // Use constant-time comparison to prevent timing attacks
      await bcrypt.compare(password, '$2b$10$invalidhashtocomparewith');
      return res.status(401).json({ error: 'Invalid credentials' });
    }
    
    // Verify password with bcrypt (slow algorithm to prevent brute force)
    const validPassword = await bcrypt.compare(password, user.passwordHash);
    
    if (!validPassword) {
      // Log failed login attempt
      Logger.warn('Failed login attempt', {
        username,
        ip: req.ip,
        userAgent: req.headers['user-agent']
      });
      
      return res.status(401).json({ error: 'Invalid credentials' });
    }
    
    // Check for account lockout
    if (user.failedLoginAttempts >= 5 && user.lockUntil > Date.now()) {
      return res.status(401).json({ error: 'Account temporarily locked' });
    }
    
    // Reset failed login attempts
    user.failedLoginAttempts = 0;
    await user.save();
    
    // Create JWT with appropriate claims and short expiration
    const token = jwt.sign(
      { 
        userId: user.id,
        role: user.role
      },
      process.env.JWT_SECRET,
      { 
        expiresIn: '15m',
        audience: 'webapp',
        issuer: 'auth-service'
      }
    );
    
    // Set token in secure, httpOnly cookie with SameSite attribute
    res.cookie('accessToken', token, {
      httpOnly: true,
      secure: process.env.NODE_ENV === 'production',
      sameSite: 'strict',
      maxAge: 15 * 60 * 1000 // 15 minutes
    });
    
    // Set longer-lived refresh token
    const refreshToken = generateRefreshToken(user.id);
    
    res.status(200).json({
      message: 'Authentication successful',
      user: {
        id: user.id,
        username: user.username,
        role: user.role
      }
    });
    
  } catch (error) {
    Logger.error('Authentication error', { error });
    res.status(500).json({ error: 'Internal server error' });
  }
}

Input Validation & Injection Protection

Proper input validation and sanitization are critical defenses against injection attacks like SQL injection, XSS, and command injection.

SQL Injection Prevention

SQL injection remains one of the most dangerous vulnerabilities. Use parameterized queries or ORM libraries to prevent it.

Vulnerable vs. Secure Implementation
Vulnerable SQL Construction (Node.js)
const userId = req.params.id;
const query = `SELECT * FROM users WHERE id = ${userId}`; // VULNERABLE!
const user = await db.query(query);
Secure SQL with Parameterized Query
const userId = req.params.id;
const query = 'SELECT * FROM users WHERE id = ?';
const user = await db.query(query, [userId]);
Framework-Specific Implementations
  • Node.js/Express: Use mysql2 or Sequelize ORM
  • Python/Django: Use Django ORM or SQLAlchemy
  • Java/Spring: Use JdbcTemplate or JPA/Hibernate
  • .NET/Entity Framework: Use LINQ or parameterized queries
  • PHP/Laravel: Use Eloquent ORM or prepared statements

Cross-Site Scripting (XSS) Prevention

XSS allows attackers to inject client-side scripts into web pages viewed by other users. Context-aware output encoding is essential.

Context-Specific Output Encoding
HTML Context:
// JavaScript/React
import { htmlEscape } from 'escape-goat';
const safeHtml = htmlEscape(userProvidedContent);

// PHP
$safeHtml = htmlspecialchars($userContent, ENT_QUOTES, 'UTF-8');

// Python/Django
{{ user_content|escape }}
JavaScript Context:
// When inserting user data into JavaScript
const safeJson = JSON.stringify(userProvidedContent);
const script = document.createElement('script');
script.textContent = `const userData = ${safeJson};`;
URL Context:
// JavaScript
const safeUrl = encodeURIComponent(userProvidedUrl);
element.href = `https://example.com/path?param=${safeUrl}`;
Content Security Policy (CSP)

Implement CSP headers to prevent execution of unauthorized scripts:

// Express.js with Helmet
app.use(helmet.contentSecurityPolicy({
  directives: {
    defaultSrc: ["'self'"],
    scriptSrc: ["'self'", "trusted-cdn.com"],
    styleSrc: ["'self'", "trusted-cdn.com"],
    imgSrc: ["'self'", "data:", "images.cdn.com"],
    connectSrc: ["'self'", "api.example.com"],
    fontSrc: ["'self'", "fonts.googleapis.com"],
    objectSrc: ["'none'"],
    baseUri: ["'self'"],
    formAction: ["'self'"],
    frameAncestors: ["'none'"]
  }
}));

Additional Injection Attack Defenses

Command Injection
  • Avoid OS commands when possible
  • Use built-in APIs instead of exec/eval
  • Implement strict allowlists if needed
  • Sanitize all user-controlled input
XML/XXE Injection
  • Disable DTDs and external entities
  • Use less complex formats (JSON)
  • Validate and sanitize XML input
  • Keep XML parsers updated
LDAP Injection
  • Sanitize input used in filters
  • Escape special characters
  • Use parameterized LDAP queries
  • Implement least privilege principle

Cryptography & Data Protection

Implement strong encryption and proper data protection measures to safeguard sensitive information.

Encryption Algorithm Selection Guide

Category Recommended Key Size Avoid Use Case
Symmetric AES-GCM, ChaCha20-Poly1305 256-bit DES, 3DES, RC4, Blowfish Data encryption at rest and in transit
Asymmetric RSA, ECC (P-256, P-384) RSA: 2048+ bits
ECC: 256+ bits		 RSA < 2048 bits, weak curves Key exchange, digital signatures
Hashing SHA-256, SHA-384, SHA-512, BLAKE2 N/A MD5, SHA-1 Integrity verification, checksums

Secure Password Storage

Never store passwords in plaintext. Use modern adaptive hashing algorithms with appropriate work factors.

Algorithm Recommendations (in order of preference):
  1. Argon2id - Memory-hard, resistant to side-channel attacks
  2. bcrypt - Battle-tested, widely supported
  3. PBKDF2 - FIPS-validated for regulatory environments
// Python Argon2 example
from argon2 import PasswordHasher

ph = PasswordHasher(
    time_cost=3,      # number of iterations
    memory_cost=65536,  # 64 MiB
    parallelism=4,    # how many parallel threads
    hash_len=32,      # the hash output size
    salt_len=16       # the salt size
)

hash = ph.hash("user_password")
try:
    ph.verify(hash, "user_password")  # True if match
    is_deprecated = ph.check_needs_rehash(hash)  # Check if needs upgrade
except VerifyMismatchError:
    # Password doesn't match
    pass

Data Encryption in Applications

Implement proper encryption for sensitive data at rest and in transit.

Key Management:
  • Use a dedicated key management service (AWS KMS, HashiCorp Vault)
  • Implement key rotation policies
  • Separate keys for different purposes/data types
  • Never hardcode encryption keys in source code
// Java AES-GCM encryption example
import javax.crypto.*;
import javax.crypto.spec.*;
import java.security.*;
import java.util.Base64;

public class EncryptionUtil {

    private static final String ALGORITHM = "AES/GCM/NoPadding";
    private static final int TAG_LENGTH_BIT = 128;
    private static final int IV_LENGTH_BYTE = 12;
    
    public static String encrypt(String plaintext, SecretKey key) throws Exception {
        // Generate random IV
        byte[] iv = new byte[IV_LENGTH_BYTE];
        new SecureRandom().nextBytes(iv);
        
        // Create GCM parameters
        GCMParameterSpec parameterSpec = new GCMParameterSpec(TAG_LENGTH_BIT, iv);
        
        // Initialize cipher for encryption
        Cipher cipher = Cipher.getInstance(ALGORITHM);
        cipher.init(Cipher.ENCRYPT_MODE, key, parameterSpec);
        
        // Encrypt
        byte[] ciphertext = cipher.doFinal(plaintext.getBytes("UTF-8"));
        
        // Combine IV and ciphertext and encode
        ByteBuffer byteBuffer = ByteBuffer.allocate(iv.length + ciphertext.length);
        byteBuffer.put(iv);
        byteBuffer.put(ciphertext);
        return Base64.getEncoder().encodeToString(byteBuffer.array());
    }
    
    // Decryption method would follow similar pattern
}

Common Cryptographic Vulnerabilities

Implementation Weaknesses
  • Using ECB mode for encryption
  • Static/hardcoded initialization vectors
  • Insufficient entropy for key generation
  • Missing integrity verification (MAC)
  • Failing to validate certificates properly
  • Mixing cryptographic primitives incorrectly
Mitigation Strategies
  • Use authenticated encryption modes (GCM, ChaCha20-Poly1305)
  • Generate cryptographically secure random IVs for each encryption
  • Use proper SecureRandom implementation
  • Implement proper certificate validation with revocation checking
  • Use high-level cryptographic libraries
  • Follow standards like NIST guidelines

Security Architecture

Implementing a robust security architecture ensures multiple layers of protection for your applications.

Defense in Depth Strategy

Implement multiple layers of security controls so that if one fails, others still provide protection.

Security Layers (from outside in):
  1. Network Security
    • Firewalls, WAF, DDoS protection
    • Network segmentation, DMZs
  2. Identity & Access Management
    • Authentication, MFA, authorization
    • SSO, identity federation
  3. Application Security
    • Input validation, output encoding
    • CSRF protection, security headers
  4. Data Security
    • Encryption, data loss prevention
    • Data classification, access controls
  5. Monitoring & Detection
    • Logging, alerting, anomaly detection
    • Intrusion detection/prevention systems

Zero Trust Architecture

Always verify and never trust. A zero trust approach assumes breaches will happen and verifies every request regardless of source.

Core Principles:
  • Verify explicitly
    • Always authenticate and authorize
    • Verify on all requests, regardless of source
  • Use least-privilege access
    • Just-in-time, just-enough access
    • Granular permissions, not broad roles
  • Assume breach
    • Minimize blast radius and segment access
    • End-to-end encryption, continuous verification

Secure by Default Configuration Example

Implementing secure default configurations in a Spring Boot application with security headers.

@Configuration
@EnableWebSecurity
public class SecurityConfig extends WebSecurityConfigurerAdapter {

    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http
            // Basic configuration
            .csrf().csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())
            .and()
            .headers()
                // Security headers
                .contentSecurityPolicy("default-src 'self'; script-src 'self' https://trusted-cdn.com; " +
                                      "style-src 'self' https://trusted-cdn.com; img-src 'self' data:; " +
                                      "connect-src 'self'; font-src 'self'; object-src 'none'; " +
                                      "media-src 'self'; frame-src 'none'; base-uri 'self'")
                .and()
                .referrerPolicy(ReferrerPolicyHeaderWriter.ReferrerPolicy.SAME_ORIGIN)
                .and()
                .frameOptions().deny()
                .and()
                .permissionsPolicy("camera=(), microphone=(), geolocation=(), interest-cohort=()")
            .and()
            // Session management
            .sessionManagement()
                .sessionCreationPolicy(SessionCreationPolicy.ALWAYS)
                .invalidSessionUrl("/login?expired")
                .maximumSessions(1)
                .maxSessionsPreventsLogin(false)
            .and().and()
            // HTTPS enforcement
            .requiresChannel()
                .anyRequest().requiresSecure()
            .and()
            // Authentication and authorization
            .authorizeRequests()
                .antMatchers("/api/public/**").permitAll()
                .antMatchers("/api/admin/**").hasRole("ADMIN")
                .antMatchers("/api/user/**").hasRole("USER")
                .anyRequest().authenticated()
            .and()
            .formLogin()
                .loginPage("/login")
                .permitAll()
            .and()
            .logout()
                .logoutUrl("/logout")
                .logoutSuccessUrl("/login?logout")
                .invalidateHttpSession(true)
                .deleteCookies("JSESSIONID");
    }
    
    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder(12);
    }
}

Secure your Express, Fastify, NestJS, and other Node.js applications against common vulnerabilities

Essential Security Packages

app.post('/user', [

    return res.status(400).json({ errors: errors.array() });
  }
  // Process valid request
});
  • Rate-limiter-flexible

    Prevent brute force and DDoS attacks with rate limiting

    const { RateLimiterMemory } = require('rate-limiter-flexible');

const rateLimiter = new RateLimiterMemory({
  points: 10, // 10 requests
  duration: 1, // per 1 second
});

app.use(async (req, res, next) => {
  try {
    await rateLimiter.consume(req.ip);
    next();
  } catch {
    res.status(429).send('Too Many Requests');
  }
});

CSRF Protection

Implement CSRF protection to prevent cross-site request forgery attacks.

const csrf = require('csurf');
const cookieParser = require('cookie-parser');

// Setup CSRF protection
app.use(cookieParser());
app.use(csrf({ cookie: true }));

// Add CSRF token to all rendered templates
app.use((req, res, next) => {
  res.locals.csrfToken = req.csrfToken();
  next();
});

// Error handler for CSRF failures
app.use((err, req, res, next) => {
  if (err.code === 'EBADCSRFTOKEN') {
    return res.status(403).json({
      error: 'CSRF token validation failed'
    });
  }
  next(err);
});

<form action="/api/submit" method="POST">
  <input type="hidden" name="_csrf" value="{{csrfToken}}">
  <!-- form fields -->
</form>

Security Tool Selection Guide

Security Need Recommended Tools When to Use
Full Security Assessment OWASP ZAP, Burp Suite, Nessus Before major releases or quarterly for critical applications
Continuous Integration SonarQube, OWASP Dependency-Check, Semgrep Every code commit or pull request
Access Control Testing Burp Suite, AuthMatrix, OWASP ZAP After changing permission systems or role definitions
Vulnerability Management OWASP DefectDojo, Threadfix, Jira Ongoing process for tracking and remediating security issues
Runtime Protection ModSecurity, OWASP CRS, CloudFlare Production environments with sensitive data or payment processing

Tool Selection Considerations

  • Size and maturity of your security program
  • Technologies used in your application stack
  • Available security skills and resources
  • Regulatory requirements and compliance needs
  • Integration capabilities with your development workflow
  • Budget constraints and open-source vs. commercial options
